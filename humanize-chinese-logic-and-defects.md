# humanize-chinese 降 AIGC 逻辑与缺陷分析

> 本文档基于对项目全部核心源码（`detect_cn.py`、`humanize_cn.py`、`restructure_cn.py`、`ngram_model.py`、`patterns_cn.json` 等）的逐行审查，系统性梳理降 AIGC 的完整逻辑链路及存在的缺陷。

---

## 一、项目概述

**humanize-chinese** 是一个纯 Python、零依赖的中文 AI 文本去痕迹工具，核心能力：

- **检测**：20+ 规则维度 + 8 项统计特征 + LR 集成评分，输出 0-100 分
- **改写**：10 步管线，6 大策略，三档自适应强度
- **验证**：Best-of-N 选优，改写后重新评分

**技术栈**：纯 Python 标准库 + 字符级 N-gram 语言模型 + 逻辑回归分类器，无神经网络、无 LLM 调用。

---

## 二、整体架构

```
用户输入文本
    │
    ▼
┌─────────────────────────────────────────────┐
│  1. 检测层 (detect_cn.py + ngram_model.py) │
│     规则匹配(20+维度) + 统计特征(8项)        │
│     → 融合评分: rule×0.2 + LR×0.8 → 0-100分 │
└──────────────────┬──────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────┐
│  2. 改写层 (humanize_cn.py + restructure_cn)│
│     三档自适应: conservative/moderate/full    │
│     6大策略管线 + best-of-N 选优              │
└──────────────────┬──────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────┐
│  3. 验证层 (detect_cn.py 再次评分)           │
│     改写后重新检测，确认降分效果              │
└─────────────────────────────────────────────┘
```

## 三、检测逻辑

### 3.1 规则检测层

**文件**：`detect_cn.py` → `detect_patterns()`
**配置**：`patterns_cn.json`

基于正则匹配扫描 20+ 维度的 AI 写作模式：

| 严重度 | 检测维度 | 示例 | 权重 |
|--------|---------|------|------|
| 🔴 Critical | 三段式套路 | 首先…其次…最后 | 8 分/条 |
| 🔴 Critical | 机械连接词 | 值得注意的是、综上所述 | 8 分/条 |
| 🔴 Critical | 空洞宏大词 | 赋能、闭环、数字化转型 | 8 分/条 |
| 🟠 High | AI 高频词 | 助力、彰显、底层逻辑 | 4 分/条 |
| 🟠 High | 填充废话 | 值得一提的是、需要指出的是 | 4 分/条 |
| 🟠 High | 模板句式 | 随着…的不断发展 | 4 分/条 |
| 🟠 High | 过度两面论 | 虽然…但是…同时 | 4 分/条 |
| 🟡 Medium | 过度谨慎 | 在一定程度上 ×5次 | 2 分/条 |
| 🟡 Medium | 列举上瘾 | ①②③④ 频繁出现 | 2 分/条 |
| 🟡 Medium | 标点过度 | 破折号/分号过多 | 2 分/条 |
| 🟡 Medium | 修辞过多 | 排比对偶过多 | 2 分/条 |
| ⚪ Style | 段落均匀 | 段落长度 CV < 0.2 | 1.5 分/条 |
| ⚪ Style | 节奏单一 | 句子长度 CV < 0.25 | 1.5 分/条 |
| ⚪ Style | 情感平淡 | 情感/个人表达密度 < 0.15% | 1.5 分/条 |
| ⚪ Style | 句首重复 | 同一句首 ×3次 | 1.5 分/条 |
| ⚪ Style | 低信息熵 | 字符熵 < 6.0 | 1.5 分/条 |

**规则层评分**：各维度加权求和，上限 60 分。

### 3.2 统计特征层

**文件**：`ngram_model.py` → `analyze_text()`

基于字符级 N-gram 语言模型（`ngram_freq_cn.json`）计算 8 项统计指标：

| 特征 | 计算方法 | AI 特征 | Cohen's d |
|------|---------|---------|-----------|
| 困惑度 (Perplexity) | trigram + 插值平滑 | 过于流畅/可预测 | 0.47 |
| 句长 CV | 句子长度的变异系数 | CV < 0.40（等长句） | **1.22** |
| 短句占比 | <10字句子比例 | < 8%（几乎无短句） | **1.21** |
| GLTR top10 | 高概率续接占比 | > 21%（选最可能的字） | 0.44 |
| DivEye skew | 惊奇度偏度 | < 1.35（缺少创造性用字） | 0.41 |
| DivEye kurt | 惊奇度峰度 | < 0.35（分布过于平均） | 0.29 |
| 逗号密度 | 逗号/百非空白字符 | < 4.5（句内停顿偏少） | -0.47 |
| 过渡词密度 | 连接词/千中文字 | > 8（然而/此外过多） | 0.62 |

**统计层评分**：各指标加权求和，上限 40 分。

### 3.3 LR 集成评分

**文件**：`ngram_model.py` → `compute_lr_score()`
**模型**：逻辑回归（`lr_coef_cn.json` / `lr_coef_academic.json` / `lr_coef_longform.json`）

```python
# 21 维特征向量 → 标准化 → 逻辑回归 → sigmoid → 0-100 分
extract_feature_vector(text)  # 提取 21 维连续特征
compute_lr_score(text, scene='general')  # 加载系数，计算 P(AI)
```

支持三种场景：
- `general`：通用文本（`lr_coef_cn.json`）
- `academic`：学术论文（`lr_coef_academic.json`）
- `novel`/`longform`：长篇小说/博客（`lr_coef_longform.json`，≥1500 字自动切换）

### 3.4 融合评分

```python
# detect_cn.py 主函数中的融合公式
score = round(0.2 * rule_score + 0.8 * lr_result['score'])
```

| 分数 | 等级 | 含义 |
|------|------|------|
| 0-24 | 🟢 LOW | 基本像人写的 |
| 25-49 | 🟡 MEDIUM | 有些 AI 痕迹 |
| 50-74 | 🟠 HIGH | 大概率 AI 生成 |
| 75-100 | 🔴 VERY HIGH | 几乎确定是 AI |

## 四、改写逻辑

### 4.1 三档自适应强度

```python
source_score = _estimate_source_aiscore(text)  # 先检测原文 AI 分

if source_score >= 25:    tier = 'full'        # 全量管线（最激进）
elif source_score >= 5:   tier = 'moderate'    # 跳过噪声注入
else:                     tier = 'conservative' # 仅短语替换（最安全）
```

**设计理由**：如果原文已经很"人味"（score < 5），全量改写反而会注入新的 AI 模式，导致分数不降反升。

### 4.2 十步改写管线

#### Step 0：预检测

调用 `detect_cn.detect_patterns()` + `calculate_score()` 给原文打分，决定改写档位。

#### Step 1：结构清理（所有档位）

**① 砍三段式** `remove_three_part_structure()`

```python
"首先，X。其次，Y。最后，Z。" → "X。再说，Y。还有，Z。"
"第一，X。第二，Y。第三，Z。" → "X。接着，Y。还有，Z。"
"其一，X。其二，Y。其三，Z。" → "X。另外，Y。还有，Z。"
```

**② 替换 AI 套话** `replace_phrases()`

先正则替换（每句最多 1 条），再纯文本替换（按长度降序）：

```python
"综上所述"     → ["总之", "说到底", "简单讲"]
"值得注意的是" → ["注意", "要提醒的是", "特别说一下"]
"不难发现"     → ["可以看到", "很明显"]
"赋能"         → ["帮助", "提升", "支持"]
"闭环"         → ["完整流程", "全链路"]
```

**选词策略（Picker）**：从候选中选困惑度**次高**的（最高常是古语/错字，最低就是 AI 原词本身）。

#### Step 2：深度句式重组（所有档位）

`deep_restructure()` 按顺序执行 7 个子步骤：

**① 句式变换** `restructure_sentences()` — 45 个正则模板

```python
# 每个模板: (正则, [候选替换1, 候选替换2])
# 匹配概率: 普通 40%, 激进 60%
# 每句最多匹配 1 个模板

"通过大数据分析，AI能够精准教学" → "AI精准教学，靠的是大数据分析"
"随着经济的不断发展，AI正在改变教育" → "AI正在改变教育，这背后是经济的持续推动"
"研究表明，AI可以提高学习效率" → "从已有研究来看，AI可以提高学习效率"
```

**② 长句拆分** `split_long_sentences()`

```python
# 在连接词处拆分 >25 字的长句
"AI不仅能够评估学生，还能提供个性化建议"
→ "AI不仅能够评估学生。AI还能提供个性化建议"
```

**③ 短句合并** `merge_short_sentences()`

```python
# 共享主语的连续短句合为一句
"研究揭示了规律。研究也验证了假设。" → "研究揭示了规律，也验证了假设。"
# 安全守卫：段落已有 20% 短句时跳过
```

**④ AI 废话删除** `remove_ai_fillers()`

```python
# 50% 概率直接删除（不是替换，是删除）
"综上所述，AI技术很重要" → "AI技术很重要"
```

**⑤ 短反应句插入** `insert_short_reactions()`

```python
# 在段落末尾插入 4-6 字短句，提升短句占比
# "确实如此。" "颇有道理。" "耐人寻味。" "值得深思。"
# 对话密度高的小说文本跳过
```

**⑥ 逗号密度补足** `boost_comma_density()`

```python
# 人类逗号密度 5.3/百字 vs AI 3.8/百字
# 在安全位置（"是一种""能够""揭示了"前）插入逗号，目标 5.0/百字
```

**⑦ 段落内信息重排** `reorder_mid_sentences()`

```python
# 4+ 句子的段落，50% 概率交换中间两句（首尾不动）
```

#### Step 3：句子合并/拆分（所有档位）

```python
merge_short_sentences(text)   # 合并 < 8 字的连续短句
split_long_sentences(text)    # 在转折连接词处拆分 > 80 字的长句
# 两个函数都有 burstiness 守卫：操作后 burstiness 下降超 20% 则回退
```

#### Step 4：节奏与多样性（所有档位）

```python
reduce_punctuation(text)        # 分号交替替换为逗号和句号
diversify_vocabulary(text)      # 同一词出现 >2 次时替换为同义词
vary_paragraph_rhythm(text)     # 合并过短/拆分过长段落（仅 moderate+）
```

#### Step 5：场景增强（仅 full 档位）

```python
add_casual_expressions(text)    # 插入"说实话""其实""讲真"等口语表达
shorten_paragraphs(text)        # 社交场景拆分长段落为 < 150 字
```

#### Step 6：低频 Bigram 注入（moderate+）

**核心策略** `reduce_high_freq_bigrams()`：

```python
# 原理：AI 文本用词高度集中（高频 bigram 多），人类用词更多样
# 策略：扫描 200+ 同义词表，对高频词做部分替换

WORD_SYNONYMS = {
    '因此': ['所以', '因而', '为此', '故而'],
    '然而': ['不过', '但', '可是', '只是'],
    '能够': ['可以', '得以', '足以', '有能力'],
    '进行': ['开展', '实施', '做', '搞'],
    '系统': ['体系', '架构', '框架'],
    # ... 共 200+ 条
}

# 替换逻辑：
# 1. 按 bigram 频率排序（频率最高的优先替换）
# 2. 选 bigram 频率"中等偏低"的候选（不是最低，最低太怪）
# 3. 只替换 60% 的出现次数（保留部分原词，避免新重复）
# 4. 多次出现时混合使用不同候选

# 场景过滤：
ACADEMIC_PRESERVE_WORDS = {'研究', '分析', '发现', '系统', '方法', ...}  # 学术核心词不替换
ACADEMIC_BLACKLIST_CANDIDATES = {'搞', '弄', '琢磨', '档次', '招数', ...}  # 过滤口语化候选
_AI_PATTERN_BLACKLIST = {'赋能', '闭环', '生态', '彰显', ...}  # 不替换成 AI 高频词本身
```

可选 `--cilin` 启用哈工大同义词词林（38,873 词）扩展候选池。

#### Step 7：噪声注入（仅 full 档位）

**① 口语化表达插入** `inject_noise_expressions()`

```python
NOISE_EXPRESSIONS = {
    'hedging':       ['说实话', '坦白讲', '客观地说'],
    'self_correction': ['或者说', '准确地讲', '换个角度看'],
    'uncertainty':   ['大概', '差不多', '似乎', '或许'],
    'transition':    ['话说回来', '反过来看', '回过头看'],
    'filler':        ['当然了', '其实', '说到底', '讲真'],
    'personal':      ['我觉得', '在我看来', '依我之见'],
}
# 每 6-7 句插入一个（density=0.15）
# 学术场景只用 hedging + self_correction + uncertainty
```

**② 句长随机化** `randomize_sentence_lengths()`

```python
# 15% 概率合并相邻短句为长句
# 15% 概率在逗号处截断长句为短句
# 目标：制造"短-长-短-特长-短"的人类节奏
# 安全守卫：不合并反应短语、不跨段落、不在归属动词/从句引导词后截断
```

#### Step 8：收尾

```python
cap_transition_density(text, target=6.0)  # 过渡词密度降到 6/千字以下
# 清理 artifact：双逗号→单逗号、双句号→单句号、多换行→双换行
```

#### Step 9：验证循环

```python
# 改写完成后，检查困惑度是否仍过低（< 350）
# 找出困惑度最低的 20% 句子，用不同种子再做一次短语替换
```

#### Step 10：Best-of-N 选优

```python
# 默认跑 10 次改写（不同随机种子），取 LR 分数最低的版本
for i in range(10):
    result = humanize(text, seed=42 + i)
    lr_score = compute_lr_score(result)
    candidates.append((lr_score, result))
return min(candidates)[2]
# --best-of-n 0 可关闭，恢复单次 5 秒速度
```

## 五、缺陷分析

### 5.1 架构级缺陷

#### 缺陷 1：改写与检测"同源作弊"

```
检测器知道什么 → 改写器就改什么
```

检测规则（`patterns_cn.json`）和改写规则（`PHRASE_REPLACEMENTS`、`_SENTENCE_TEMPLATES`）是同一批人手工编写的。改写器本质上是"对着答案改卷子"：

- 检测器说"综上所述"是 AI 词 → 改写器就把"综上所述"换成"总之"
- 检测器说句长 CV < 0.40 是 AI → 改写器就随机化句长
- 检测器说短句占比 < 8% 是 AI → 改写器就插入短反应句

**影响**：本项目只能骗过自己的检测器。换一个检测器（知网、维普、GPTZero），这些改写可能完全无效。README 中也承认："CNKI/维普/万方没有公开 API，我们无法接入作为 oracle"。

#### 缺陷 2：没有语义保持机制

整个改写管线中，没有任何步骤检查"改写是否改变了原意"。可能出问题的场景：

- "应用" → "施用" → "施用身份证件"（语义错误）
- "系统" → "架构" → "神经系统架构"（语义变了）
- "有效"被跳过，但其他词替换可能破坏上下文

项目用手工黑名单 `ACADEMIC_BLACKLIST_CANDIDATES` 来缓解，但这是"打地鼠"式修复，无法覆盖所有情况。

#### 缺陷 3：Best-of-N 选优的过拟合风险

默认跑 10 次取 LR 分数最低的版本，但 LR 模型就是本项目自己训练的（`lr_coef_cn.json`）。这等于：用模型 A 训练 → 用模型 A 评分 → 选模型 A 打分最低的版本。在 HC3 数据集上效果好，不代表在真实场景中效果好，因为 LR 系数是在 HC3 上拟合的，改写策略也是针对 HC3 校准的。

### 5.2 策略级缺陷

#### 缺陷 4：困惑度引导选词（Picker）的理论缺陷

```python
def pick_best_replacement(sentence, old, candidates):
    scored.sort(key=lambda x: x[1])
    return scored[-2][0]  # 选次高困惑度的候选
```

- 困惑度 ≠ 自然度。一个词困惑度高可能是因为它根本不通顺，而不是因为它"更像人写的"
- 字符级 trigram 模型的困惑度非常粗糙，无法区分"少见但正确"和"少见且错误"
- 选次高而非最高是经验规则，没有理论依据

#### 缺陷 5：噪声注入可能适得其反

- 学术文本中插入"说实话"会降低学术性（已有 `NOISE_ACADEMIC_CATEGORIES` 过滤，但不完美）
- 这些表达本身就成了新的"模式"——如果检测器把"说实话""怎么说呢"当作 AI 伪装特征，改写反而会提高分数
- 插入位置是随机的，不考虑语义连贯性："量子力学，说实话，是一种物理理论。" 读起来很怪

#### 缺陷 6：短反应句插入的机械感

```python
_SHORT_REACTIONS_NEUTRAL = [
    '确实如此。', '颇有道理。', '不无道理。',
    '事出有因。', '耐人寻味。', '值得深思。',
    ...
]
```

这些反应句本身就成了新的"模式"。如果一段文本末尾反复出现"确实如此。""值得深思。"，一个足够聪明的检测器会把它当作"反检测模式"来识别。

#### 缺陷 7：同义词替换不感知词性

```python
WORD_SYNONYMS = {
    '重要': ['关键', '核心', '要紧', '紧要'],
}
```

"重要"可以是形容词（重要的人）也可以是名词（重要性）。替换为"关键"时："重要意义" → "关键意义" ✅，但边界情况会出错。没有词性标注，纯字符串替换。

### 5.3 工程级缺陷

#### 缺陷 8：正则模板的脆弱性

45 个模板全部基于正则匹配：

- 只能匹配已知的句式。"通过X，Y能够Z"能匹配，"借助X，Y得以Z"匹配不到（语义相同）
- 正则边界用 `[^，,。]` 做分隔，遇到嵌套标点会匹配错误
- 模板之间可能冲突，代码限制每句只匹配一个模板，也导致覆盖率下降

#### 缺陷 9：随机性导致不可控

整个管线大量使用 `random.random()` 做概率决策（句式变换 40%/60%、长句拆分 50%、短句合并 40%、废话删除 50%、信息重排 50%、噪声插入 15%）。同一个文本不同 seed 可能产生质量差异巨大的结果。`best_of_n=10` 缓解了这个问题，但代价是 10 倍延迟，而且选的是"LR 分数最低"的，不一定是"读起来最自然的"。

#### 缺陷 10：段落边界处理不一致

代码中反复出现段落边界处理（`merge_short_sentences`、`randomize_sentence_lengths`、`inject_noise_expressions`、`cap_transition_density` 都有各自的段落守卫逻辑），说明这是一个反复踩坑的地方。每个函数都在独立处理段落边界，没有统一的段落感知抽象，容易出现边界情况 bug。

### 5.4 数据级缺陷

#### 缺陷 11：训练数据单一

- LR 分类器主要基于 HC3-Chinese（ChatGPT/GPT-3.5 的数据），没有 GPT-4o/Claude/Gemini
- 新一代 LLM 的写作风格与 GPT-3.5 差异很大
- README 中承认："默认模式对现代 LLM 的长篇创作欠估 ~15 分"

#### 缺陷 12：N-gram 频率表的局限

- 频率表是静态的，不随语言演变更新
- "赋能""闭环"等词在 2024 年已进入日常用语，但频率表中可能仍标记为低频
- 不同领域的文本（医学、法律、文学）用词分布差异大，单一频率表不够

### 5.5 评估级缺陷

#### 缺陷 13：自评自改，缺乏独立验证

```
评估闭环：
  HC3 数据 → 训练 LR → 训练阈值 → 训练改写策略 → 在 HC3 上评测 → 报告 95.5%
```

整个闭环都在 HC3 这一个数据集上完成，没有在知网/维普/万方真实检测上验证，没有在 GPT-4o/Claude/Gemini 生成文本上验证，没有在人类判定的盲测上验证。

#### 缺陷 14：评测指标单一

只报告了 AI vs 人类分数差距（gap）、正确分离率（correct rate）、改写后降分幅度（delta），没有报告：
- 改写后的可读性/流畅度评分
- 改写后的语义保持率
- 人类盲测通过率
- 跨检测器泛化性

## 六、短文本能力评估

### 短文本定义

短文本通常指 < 100 中文字符（微博、标题、摘要、问答对等）。

### 检测能力

| 功能 | 最低长度要求 | 短文本能否触发 |
|------|------------|--------------|
| 规则匹配 | 无限制 | ✅ 正常工作 |
| 情感平坦度 | > 300 字 | ❌ 完全跳过 |
| 段落均匀度 | ≥ 3 段落 | ❌ 完全跳过 |
| 句长 burstiness | > 5 句子 | ❌ 完全跳过 |
| N-gram 统计特征 | ≥ 100 字 | ❌ 核心统计层完全关闭 |
| GLTR rank 分桶 | ≥ 30 字 | ❌ 完全跳过 |
| DivEye 惊奇度 | ≥ 16 log_probs | ❌ 完全跳过 |

| LR 集成评分 | 无显式限制 | ⚠️ 21 维特征几乎全为 0 |

## 八、改进方向建议

### 最小改动（不改架构）

1. **短文本专用评分**：`char_count < 100` 时用纯规则评分，不融合 LR
2. **降低统计阈值**：困惑度 3 字起算、GLTR 10 字起算
3. **多样化反应句**：扩充短反应句池，避免模式化

### 中等改动（引入轻量模型）

1. **BERT 做 Picker**：用 BERT MLM 概率替代 trigram perplexity 选词
2. **BERT 做语义保持检查**：改写前后 embedding 相似度 < 0.85 则回退
3. **BERT 增强检测**：微调 BERT 分类器替代或补充 LR 评分

### 大改动（架构升级）

1. **独立第三方验证**：接入 PaperPass/朱雀等真实 AIGC 检测 API
2. **依存句法分析**：用 `ddparser`/`spaCy` 替代正则模板，自动识别主谓宾
3. **生成式改写**：用 BERT masked LM 或小参数生成模型做真正的语义改写
4. **人类盲测**：收集改写前后的文本，让人类评判哪个更像人写的
